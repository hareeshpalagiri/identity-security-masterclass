## **Chapter 06 — Identity Security Blueprint**

*One complete end‑to‑end identity security reference architecture for enterprise environments*

***

## 🔰 1) Introduction

The **Identity Security Blueprint** provides a holistic end‑to‑end design for securing modern enterprise identity environments that span:

*   **On‑prem Active Directory**
*   **Microsoft Entra ID (Azure AD)**
*   **Hybrid identity connectors (Entra Connect / Cloud Sync)**
*   **SaaS applications**
*   **Privileged access infrastructure**
*   **Monitoring & detection ecosystems**

This blueprint acts as a **reference architecture** and **implementation guide** for building a Zero‑Trust, identity‑first security model.

***

## 🧠 2) Identity Security Principles (Foundation)

A secure identity architecture must follow these core principles:

### **1. Verify Explicitly (Continuous Authentication)**

*   Verify user identity
*   Verify device health
*   Verify network, risk & session context

### **2. Least‑Privilege Access (Minimize Standing Privileges)**

*   Role-based access
*   Just‑In‑Time (JIT) elevation
*   Just‑Enough‑Access (JEA)
*   No permanent admin roles

### **3. Assume Breach**

*   Monitor everywhere
*   Detect anomalous identity behavior
*   Limit lateral movement
*   Token & session protection

These principles form the baseline for the rest of the blueprint.

## 🏗️ 4) Active Directory Security Blueprint

### **A) AD Hardening Essentials**

*   Disable LM/NTLMv1
*   Enforce LDAP signing & channel binding
*   Harden Domain Controllers (Tier‑0)
*   Deploy LAPS/gMSA
*   Enforce SMB signing
*   Block anonymous access

### **B) Credential & Ticket Protection**

*   Enable Credential Guard
*   Enable Protected Users group
*   Mitigate:
    *   Kerberoasting
    *   AS‑REP roasting
    *   Golden Ticket
    *   DCSync
*   Monitor Event IDs: 4768, 4769, 4624, 4662

### **C) Privileged Access Controls**

*   Use a tiered model:
    *   **Tier 0** — DCs, identity systems
    *   **Tier 1** — servers, apps
    *   **Tier 2** — user workstations
*   Domain Admin access only through PAW
*   No domain admin login on Tier1/2

***

## 🌩️ 5) Entra ID Security Blueprint

### **A) Conditional Access Baseline**

*   Require MFA
*   Block legacy authentication
*   Require compliant or hybrid-joined device
*   Block risky sign-ins
*   Limit admin access to secure locations

### **B) Identity Protection Policies**

*   User Risk = **High → Block**
*   Sign‑in Risk = **Medium/High → MFA**
*   Risk-based password reset workflow

### **C) Application Governance**

*   Disable user consent
*   Use verified publisher apps
*   Rotate certificates, eliminate secrets
*   Remove stale enterprise applications

### **D) Privileged Identity Management**

*   Admin roles JIT
*   Approval workflow
*   Alerts on elevation
*   High‑impact roles:
    *   Global Admin
    *   Privileged Auth Admin
    *   App Admin
    *   Cloud App Admin

### **E) Zero‑Trust Token Protection**

*   Enable CAE
*   Limit refresh token lifetime
*   Block token issuance for risky devices

***

## 🔗 6) Hybrid Identity Blueprint (AD + Entra)

### **Sync Layer**

*   Prefer **PHS**
*   PTA only when required
*   Cloud Sync for multi‑forest estates
*   Protect Entra Connect server as Tier‑0

### **Device Layer**

*   Hybrid Azure AD Join for legacy estates
*   Azure AD Join for cloud-first
*   Entra ID device registration
*   Intune device compliance policies

### **SSO Layer**

*   Use Seamless SSO
*   Replace ADFS where possible
*   Migrate SAML Relying Parties → Entra ID

***

## 🔐 7) Attack Path Mitigation Blueprint

### **1. Password Spray**

*   Block legacy auth
*   Smart lockout
*   CA risky sign‑in blocking

### **2. Kerberoasting**

*   Rotate service account passwords (gMSA)
*   Enforce AES
*   Monitor 4769 spikes

### **3. Token Replay**

*   Enable CAE
*   Short token lifetime policies
*   Device-based access requirements

### **4. OAuth Consent Phishing**

*   Disable user consent
*   Admin approval workflow
*   Monitor app permissions

### **5. Golden Ticket / DCSync**

*   Protect Domain Controllers
*   Monitor 4662 events
*   Restrict replication permissions

## 🛠️ 9) Incident Response Workflow (Identity‑Focused IR)

    1. Alert triggered (Defender for Identity / Identity Protection / Sentinel)
    2. Contain user/device
       - Block sign-in
       - Revoke sessions
       - Force password reset
       - Remove app consents
    3. Investigate identity timeline
       - Sign-in logs
       - Token anomalies
       - Device compliance
    4. Assess lateral movement (AD + Entra)
    5. Validate privilege escalation
    6. Recover
       - Reset accounts
       - Rotate secrets/certs
       - Clean up ADFS/SPN issues
    7. Strengthen policies to prevent recurrence
